Install & Init

从安装 Plugin 到 Creator 准备好开始开发，经历的每一步。精确到可直接作为 Plugin 开发规格。

Dev & Publish 流程 →
Overview

阶段总览

阶段名称频率说明
Phase 1 安装 Plugin 仅一次 从开发者平台获取安装指令，在 Claude Desktop 中安装
Phase 2 首次初始化 每个新 Skill 一次 登录 + 创建 Skill 项目 + 打开项目文件夹
Phase 3 日常进入开发 每次开发 打开项目文件夹 → Plugin 自动激活 → 就绪
设计原则：Phase 1 和 Phase 2 的摩擦由 Phase 3 的丝滑来偿还。初始设置的每一步都是为了让日常开发体验最优——完整的 Claude Code 项目级能力（CLAUDE.md 自动加载、Memory 持久化、Git 集成）只有在 Creator 正确打开项目文件夹时才能生效。
环境假设：流程优先基于 Claude Desktop 的 Code 模块设计，后续实现时增加对 CLI 的支持。先以 Claude Code 为基础开发，后续增加对 Codex 的支持。
核心准则：低技术门槛，不低估用户。Plugin 的目标用户中大部分没有技术背景，但也有一部分具备开发经验。Plugin 的默认交互应做到：不需要理解底层技术原理，照着提示就能完成操作。同时避免过度解释——技术复杂度由 Plugin 吸收，但不要让懂技术的 Creator 觉得被当成小白。具体做法：用通俗语言作为默认表达，技术细节按需展示而非强制隐藏。
双触发模式：Plugin 的所有功能同时支持斜杠命令和自然语言触发。例如 /minus publish 和"帮我发布"等价，/minus 和"打开 Minus"等价。Creator 不需要记住任何命令，用自然语言描述意图即可。斜杠命令作为快捷方式保留，供熟悉的 Creator 使用。
上下文窗口管理：Creator 不会主动管理会话上下文，Plugin 必须内置容量检查机制。每次完成一个主要任务后（如完成一个节点的开发），Plugin 自动评估当前会话的上下文使用情况。当接近上限时，Plugin 主动将当前进度保存到 Memory，然后引导 Creator 开启新 Session 继续工作，并给出完整的操作步骤。Creator 不需要理解"上下文"是什么概念，Plugin 负责在合适的时机平滑切换。

2

首次初始化

每个新 Skill 一次 · 约 3-5 分钟

Phase 2 分为两个子流程：登录（仅首次）和创建项目（每个新 Skill）。Plugin 自动判断当前状态，只执行必要的步骤。

Step 2.0

Plugin 自动进入初始化

Plugin

安装完成后（Phase 1），Plugin 在当前 Session 中直接继续，自动进入初始化流程。Creator 无需输入任何额外指令——从安装到登录是一个连贯的体验。

后续创建新 Skill：首次初始化完成后，如果 Creator 想在新 Session 中创建另一个 Skill 项目，可通过 /minus 或自然语言（如「我想创建新 Skill」）手动触发。日常开发中打开已有项目时，Plugin 通过 SessionStart hook 自动激活。
Step 2.1

Plugin 自动检查鉴权状态

Plugin

Plugin 读取平台标准配置目录下的凭证文件：

平台凭证路径说明
Mac~/.minus/credentials.json点号开头，Finder 默认隐藏
Windows%APPDATA%\Minus\credentials.json即 C:\Users\{用户}\AppData\Roaming\Minus\
Linux~/.config/minus/credentials.jsonXDG 规范

判断分支

有 token 且未过期 → 静默通过，Creator 无感知。直接进入 2.2。

有 token 但过期 → 自动尝试 refresh。成功则静默通过；失败则进入 2.1a。

无 token → 进入 2.1a。

凭证文件内容（两种 auth_type 共用同一文件）

credentials.json — API Key 模式
{
  "auth_type": "api_key",
  "api_key": "mk_abc123...",
  "user_id": "usr_abc123",
  "display_name": "Creator Name",
  "email": "creator@example.com"
}
credentials.json — OAuth 模式
{
  "auth_type": "oauth",
  "user_id": "usr_abc123",
  "display_name": "Creator Name",
  "email": "creator@example.com",
  "access_token": "eyJhbG...",
  "refresh_token": "dGhpcyB...",
  "expires_at": "2026-06-15T00:00:00Z"

文件权限设为 600（仅 owner 可读写）。

Plugin 自动检查项目上下文

Plugin

鉴权通过后，Plugin 检查当前工作目录的内容。目录内容是权威判断依据，projects.json 仅为便利缓存。

判断分支

当前目录存在 .minus/skill.json → 有效 Skill 项目。如不在 projects.json 中则静默追加注册。直接跳到 Phase 3。

当前目录与 Minus 无关 → 进入 2.3。同时读取 projects.json，如有已注册项目则在选项中展示。

Step 2.4

Scaffold 项目结构

Plugin

平台注册与凭证获取

在创建文件之前，Plugin 先调用 Minus 平台 API 注册新 Skill：

1

Plugin 调用 POST /api/v1/skills 接口，传入 Creator 信息（user_id、display_name）

2

平台返回两个关键字段：
Skill ID（如 sk_8f3a2b）：Skill 的唯一标识
Skill API Key（如 sk_key_xxx...）：Skill 运行时调用平台数据 API 的凭证

3

Plugin 将 Skill ID 写入 .minus/skill.json，Skill API Key 写入 .env.local

凭证归属用途存储位置
Creator 登录凭证 Creator 个人 开发时操作：创建项目、发布、管理 Skill ~/.minus/credentials.json
Skill API Key 单个 Skill 运行时操作：用户使用 Skill 时调用平台数据 API 项目目录 .env.local（MINUS_AI_SKILL_API_KEY）

分离的原因：避免 Creator 个人凭证暴露在 Skill 运行环境中，且每个 Skill 可独立控制数据 API 的访问权限和用量。

离线场景：如果此时网络不可用，.minus/skill.json 中 skillId 字段标记为 "pending"。下次 Creator 在该项目中执行 /minus 且网络可用时，Plugin 自动完成注册并回写。在注册完成之前，本地开发不受影响，仅发布功能和需要调用平台数据 API 的步骤测试被阻塞。

自定义路径分支

如果 Creator 回复了一个自定义路径，Plugin 执行：

1

2

在新 Workspace 下重新 scaffold 项目

3

更新 config.json 中的 workspace_dir 为新 Workspace 路径

4

更新 projects.json 中该项目的路径

5

原默认路径下的空项目文件夹不做处理，留着无影响

6

重新输出引导消息，指向新路径（不再提供"放在其他位置"选项，避免循环）

为什么必须手动打开文件夹

Claude Desktop 的 Code 模块只有在正确设置工作目录时，才能激活项目级功能：

  • CLAUDE.md 只有在工作目录下才会被自动加载
  • Memory 的项目级持久化依赖工作目录
  • Git 操作需在项目目录下执行
  • 文件读写路径以工作目录为基准

跳过这一步，Creator 会失去 Claude Code 最有价值的项目级能力，长期开发体验大幅下降。

降低摩擦的辅助手段

  • 方案 A：消息中包含可点击路径链接（如 Desktop 支持）
  • 方案 B：自动将路径复制到剪贴板
  • 方案 C：未来如 Desktop 支持 API 切换工作目录，可直接自动切换

Creator

3

日常进入开发

每次开发 · 目标 3-5 秒就绪

Phase 3 是 Creator 最频繁经历的流程。设计目标：从打开项目到进入开发状态 3-5 秒。

Step 3.1

Creator 打开项目文件夹 → Plugin 自动激活

Creator

Creator 启动 Claude Desktop，将 Skill 项目文件夹设为工作目录。Claude Code 自动：

  • 加载 CLAUDE.md
  • 加载项目级 Memory
  • SessionStart hook 检测到 .minus/skill.json，自动激活 Plugin
无需手动输入 /minus。Plugin 通过 SessionStart hook 检测到当前目录是 Minus Skill 项目后，自动进入开发模式。/minus 作为手动备用入口保留，供 Creator 在非项目目录中主动触发。
Step 3.2

Plugin 执行环境检查

Plugin

Plugin 依次执行四项检查。前两项静默完成；第三、四项视情况告知 Creator。

① 鉴权检查（静默）

读取凭证文件，校验是否已登录 Minus Plugin、API Key 是否有效。过期则自动 refresh。refresh 失败则触发重新登录流程。

② 项目识别（静默）

检查当前目录是否存在 .minus/skill.json。存在但不在 projects.json 中则静默追加注册。不存在则提示可能打开了错误文件夹。

③ 开发环境初始化（首次）/ 依赖检查（后续）

Principles

核心设计原则

凭证与项目分离

凭证在平台配置目录（隐藏），项目在 Workspace 目录（可见）

目录检测为权威

.minus/skill.json 存在即有效，projects.json 仅是便利缓存

登录一次性

Token 持久化 + 自动 refresh，Creator 通常只登录一次

Workspace 路径不问

安装时默认创建，创建首个项目时可在对话中自定义，旧路径不处理

最少手动操作

唯一手动步骤：在 Desktop 打开项目文件夹。安装后自动进入初始化，日常开发通过 SessionStart hook 自动激活，全程无需手动输入 /minus

上下文智能引导

Creator 在任何位置提出任何请求，Plugin 都不拒绝、不说教，能做就做，做不了的引导到正确位置

Desktop Code 模块优先

不依赖 cd、终端操作等 CLI 思维
