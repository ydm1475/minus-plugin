Creator Plugin Flow
Dev & Publish

Skill 项目就绪后，Creator 的开发与发布全流程。精确到可直接作为 Plugin 开发规格。

前置文档：Install & Init →
Phase Overview

阶段总览

起点：Creator 已完成 Phase 3（日常进入），Skill 初始页面已生成并打开，开发环境就绪。

阶段 名称 频率 说明
Phase 1~3 安装 & 初始化 & 日常进入 — 见 Install & Init
Phase 4 Skill 开发 持续 结构设计 → 逐节点开发 → 结果呈现 → 输出定义
Phase 5 发布上线 每个版本一次 校验 → 打包 → 上传 → 审核 → 预览 → 正式发布
环境假设：流程优先基于 Claude Desktop 的 Code 模块设计，后续实现时增加对 CLI 的支持。
核心准则：零技术门槛。Plugin 的目标用户是没有技术背景的 Creator。在整个流程中不要假设 Creator 知道任何技术层面的东西——什么是端口、什么是 API、什么是 YAML、什么是依赖安装，这些对 Creator 来说都是陌生概念。Plugin 的所有说明和指引必须做到：Creator 不需要理解背后的技术原理，照着提示就能完成操作。技术复杂度由 Plugin 吸收，不暴露给 Creator。
双触发模式：Plugin 的所有功能同时支持斜杠命令和自然语言触发。例如 /minus publish 和"帮我发布"等价，/minus 和"打开 Minus"等价。Creator 不需要记住任何命令，用自然语言描述意图即可。斜杠命令作为快捷方式保留，供熟悉的 Creator 使用。
上下文窗口管理：Creator 不会主动管理会话上下文，Plugin 必须内置容量检查机制，贯穿整个开发过程持续检查，不限于节点边界。当接近上限时，Plugin 主动将当前进度保存到 Memory，然后引导 Creator 开启新 Session 继续工作，并给出完整的操作步骤。Creator 不需要理解"上下文"是什么概念，Plugin 负责在合适的时机平滑切换。
Platform Capabilities

平台能力

Minus 平台为所有 Skill 提供三类系统级基础设施：代码层的内置工具库、界面层的 UI 组件规范、会话层的上下文管理。这些能力由平台统一维护，Creator 无需自己实现，保证所有 Skill 的质量底线一致。

内置工具库

Plugin 提供一套标准工具包，大模型在生成 Skill 代码时直接调用。保证所有 Skill 的输出格式统一、质量稳定，避免大模型每次自己写重复的基础逻辑。

格式化工具

方法 用途 示例
minus.format.number(value) 数字千分位 1234567 → 1,234,567
minus.format.decimal(value, digits) 保留小数位 3.14159, 2 → 3.14
minus.format.percent(value) 百分比 0.156 → 15.6%
minus.format.currency(value, currency) 货币 29.9, 'USD' → $29.90
minus.format.date(value, pattern) 日期 '2026-05-17', 'MM/DD' → 05/17
minus.format.compact(value) 缩写大数字 1580000 → 1.58M

数据处理工具

方法 用途
minus.data.sort(list, field, order) 按字段排序
minus.data.filter(list, condition) 条件过滤
minus.data.group(list, field) 按字段分组
minus.data.dedupe(list, field) 按字段去重
minus.data.top(list, field, n) 取前 N 项

输出工具

方法 用途
minus.output.table(data, columns) 渲染数据表格
minus.output.card(title, value, change) 渲染指标卡片
minus.output.excel(data, columns, filename) 生成 Excel 下载
minus.output.html(content, filename) 生成 HTML 报告下载

扩展原则：当大模型发现某个格式化/处理逻辑在多个 Skill 中重复出现时，应提议将其沉淀为新的内置方法。工具库随平台迭代持续扩展。

Skill UI 组件规范

Minus 提供一套标准 UI 组件，所有 Skill 共享同一界面框架。Creator 只配置组件的内容，不控制组件的结构和样式。这保证了两件事：用户在不同 Skill 之间切换时体验一致；Creator 无需关心界面设计，专注于业务逻辑。

Pipeline 进度条

固定 步骤条布局、进度动画、当前 / 完成 / 未开始三种状态图标
Creator 配置 步骤名称、步骤数量

用户看到的始终是统一的步骤进度条，清楚知道当前在哪一步、还有几步。无论 Skill 有 2 步还是 10 步，视觉体验完全一致。

输入表单

固定 表单布局、输入框样式、校验提示、提交按钮
Creator 配置 输入项的 label、placeholder、是否必填、输入数量（单个 / 多个）

支持的输入类型：

  • 单行文本（关键词、ASIN 等）
  • 多行文本（批量输入，如多个 ASIN）
  • 选择器（下拉选项）
  • 数字输入

确认弹窗

固定 弹窗结构、按钮位置（确认 / 取消）、遮罩层
Creator 配置 提示文案、确认按钮文字

用于 pipeline 中 requires_confirmation: true 的步骤。用户看到当前步骤的输出后，通过确认弹窗决定是否继续下一步。

结果页

固定 页面骨架：摘要区（顶部）→ 数据区（中部）→ 下载区（底部）
Creator 配置 摘要区内容、数据区的表格列 / 卡片指标、下载区的文件列表

三个区块的位置和布局不可更改，Creator 只定义每个区块展示什么内容。

加载状态

固定 加载动画、步骤名称显示、预估时间提示
Creator 配置 无（自动从 pipeline 定义中读取步骤名称）

每个步骤执行时，用户看到统一的加载界面，包含当前步骤名称和进度。Creator 不需要为加载状态做任何配置。

错误提示

固定 错误卡片样式、重试按钮、返回按钮
Creator 配置 错误描述文案模板（可选，默认由系统生成通用提示）

扩展原则：组件库随平台迭代持续扩展。当多个 Skill 出现相似的交互模式时，应提炼为新的标准组件。Creator 不能自定义组件外观，但可以向平台提出组件需求。

上下文窗口管理

Creator 不会主动管理会话上下文。Plugin 内置容量检查机制，贯穿整个开发过程持续检查，不限于节点边界——单个节点也可能耗尽一个 Session。正常状态下不打断 Creator；当接近上限时执行以下流程。

保存进度

Plugin 将当前开发状态写入 Memory，包括：

  • 已完成的步骤及其状态
  • 当前正在进行的任务及进展
  • 接下来需要继续的工作
  • Creator 在本轮会话中表达的偏好和决策

通知并引导切换

上下文切换
Plugin 当前对话的记忆空间快满了，继续工作可能会影响质量。
我已经把你的进度保存好了：
✓ 步骤 1「关键词数据采集」已完成
✓ 步骤 2「竞争度分析」开发中（数据需求已确认，处理逻辑待定）
请开一个新对话继续：
1. 点击左上角的 ＋ 开始新对话
2. 输入 /minus 或"打开 Minus"
3. Plugin 会自动读取进度，从中断的地方继续
措辞原则：不要出现"上下文窗口""token""context window"等技术术语。用 Creator 能理解的说法——"记忆空间""对话容量"等。
为什么不自动切换？Claude Desktop 中 Session 必须由用户手动创建，Plugin 无法自动开启新 Session。Plugin 能做的是让切换尽可能简单——保存好进度，给出明确的操作步骤，Creator 照做即可。
进度恢复：新 Session 启动后，Plugin 通过 /minus 进入项目（Phase 3 流程），自动读取 Memory 中的进度记录，主动告诉 Creator"上次做到哪里了，接下来继续什么"，无需 Creator 自己回忆。
4

Skill 开发

持续 · Creator 花时间最多的阶段
Plugin 在这里的角色是引导者——帮 Creator 把一个模糊的想法变成可运行的 Skill。所有实际的开发工作（写 prompt、调 API、处理输出）都由 Claude Code 的原生能力完成，Plugin 只编排流程。
Step 4.1

Creator + Plugin

Plugin 采用逐步引导的方式，一次只聚焦一个问题，确保 Creator 把每个环节都想清楚：

Step 4.2

逐节点开发

Creator + Plugin

Plugin 引导 Creator 按顺序开发每个 pipeline 节点。每个节点的开发是一轮完整的对话，Plugin 依次引导 Creator 确认以下四个维度：

① 数据需求：完成这一步需要什么数据？

Plugin 通过数据服务商提供的 MCP 自动发现可用的 API 列表，向 Creator 推荐匹配当前节点需求的接口。Creator 不需要自己去查 API 文档。

数据需求对话
Plugin 现在开发步骤 1「关键词数据采集」。
我通过 MCP 查询了可用的数据接口，以下 API 与这个步骤相关：
· market_get_keyword_demand — 关键词搜索量、点击量、购买率
· market_get_keyword_history — 关键词 12 个月趋势数据
· market_get_keyword_competition — 关键词竞争密度指标
你需要哪些？还有其他数据需求吗？
Creator 前两个就够了，搜索量和趋势

MCP 发现机制：所有数据服务商在 Minus 平台上注册时，会提供一个 MCP（Model Context Protocol）接口用于查询其全部可用 API。Plugin 在开发阶段通过 MCP 动态获取 API 目录，而非依赖静态文档。数据服务商新增 API 后，Creator 在下一次开发时就能自动看到。

直接透传原始数据？做聚合/排序？用大模型做分析总结？

代码编写、调试与测试：贯穿节点开发全程。Plugin 在每个维度确认后即时编写和调试代码，Creator 随时可在浏览器中体验效果并提出修改意见。测试不是独立环节，而是持续进行的。

统一预览地址：Creator 只需要记住一个地址来测试 Skill。Plugin 在首次启动预览时自动检测可用端口，如果该端口被占用，Plugin 自动切换到新端口并通知 Creator。无论底层有多少服务，Creator 看到的始终是一个入口：http://localhost:{port}。

Step 4.3

结果呈现设计

Creator + Plugin

所有 pipeline 节点开发完成后，Plugin 引导 Creator 定义 Skill 的最终输出——这是终端用户看到的最后一屏，是独立于任何 pipeline 节点的呈现层。

Plugin 引导 Creator 逐一确认以下两个方面：

① 结果摘要：如何总结这个 Skill 的运行结果？

Skill 运行结束后，结果页底部会有一段摘要来总结分析结论。Plugin 先展示各步骤产出的数据全景，再引导 Creator 选择由大模型在运行时基于实际数据自动生成摘要，或由 Creator 定义模板。如果选择大模型，Plugin 会结合当前 Skill 的实际数据和 Creator 的业务目标动态生成确认内容，得到明确确认后再继续。

Step 4.4

输出 Schema 定义（v1 预留）

Plugin
v1 暂不支持 Skill 链路对接，此步骤仅预留数据结构。

当一个 Skill 的输出可以作为另一个 Skill 的输入时，需要定义标准化的输出 Schema。Plugin 在后端预留 output_schema 字段：

后端存储的输出 Schema（v1 预留） 
output_schema:                    # v1 预留，暂不启用
  type: keyword_analysis
  fields:
    - name: keyword
      type: string
    - name: search_volume
      type: number
    - name: competition_score
      type: number
    - name: recommended_keywords
      type: array

Plugin 在 4.3 完成后自动根据结果数据生成 output_schema 草稿，通过平台 API 写入后端，但不对 Creator 展示（避免增加 v1 的认知负担）。后续版本启用 Skill 链路功能时，这个 Schema 是对接的基础。

Phase 4 完成。 Skill 开发完毕，可以进入发布流程。

Step 5.4

审核

Minus 团队

审核由 Minus 团队在平台侧完成，Plugin 不参与。

审核维度 说明
功能完整性 Skill 是否能正常运行，步骤衔接是否顺畅
内容合规 不含违规内容，符合平台使用条款
质量标准 输出质量是否达到可发布水准
安全性 数据调用是否在授权范围内，无越权操作

审核结果通知

  • 通过 → 页面出现「预览」和「发布」两个按钮
  • 未通过 → 列出失败原因和整改意见，无任何按钮，修改后重新 /minus publish
Step 5.5

预览版 → 正式发布

Creator

审核通过后，Skill 先以预览版形式上线。Creator 可以在 Minus 平台上自行测试预览版，确认无误后在开发者平台点击"正式发布"。

正式发布后，Skill 对所有 Minus 用户可见。
