# Plugin 守护能力（Creator Guardian）

> Creator 不需要理解任何技术概念，Plugin 负责吸收所有技术复杂度。本文档定义 Plugin 的守护能力体系——每一项能力解决一类"Creator 不知道、不应该知道、也不需要知道"的技术问题。
>
> **关联文档：** 安装与初始化流程见 [8 - Install & Init.md](8%20-%20Install%20%26%20Init.md)，开发与发布流程见 [9 - Dev & Publish.md](9%20-%20Dev%20%26%20Publish.md)。

---

## 设计哲学

Plugin 的目标用户中大部分没有技术背景。他们不知道什么是目录结构、环境变量、热更新、Git、依赖管理——**他们也不需要知道**。

传统开发工具的设计假设用户理解这些概念，在出问题时抛出技术错误信息，期望用户自行排查。Plugin 的设计假设相反：

| 传统工具 | Plugin |
|---------|--------|
| 报错让用户自己修 | 自动诊断并修复，修不了才告知 |
| 期望用户理解技术概念 | 用业务语言沟通，技术细节内部消化 |
| 用户主动管理环境 | Plugin 自动管理，用户无感知 |
| 出错时给日志 | 出错时给行动方案 |

**核心原则：**

| 原则 | 说明 |
|------|------|
| 不拒绝 | Creator 的意图永远合理，Plugin 解决"怎么做"的问题 |
| 不说教 | 不解释技术原理，直接给结果或行动方案 |
| 能做就做 | 能自动完成的绝不询问，减少 Creator 的决策负担 |
| 出错给方案 | 不给原始错误日志，给"发生了什么 + 怎么解决"  |
| 渐进展示 | 默认隐藏技术细节，Creator 追问时再展开 |

## 能力总览

| # | 能力 | 解决什么问题 | 状态 |
|---|------|------------|------|
| 1 | 目录上下文智能引导 | Creator 不懂"目录""Session"的概念，在任何位置提任何请求 | 已设计 |
| 2 | 自动环境管理 | Creator 不知道改了代码需要重启服务 | 已设计 |
| 3 | 上下文窗口管理 | Creator 不知道对话有长度限制，会话突然"失忆" | 已设计 |
| 4 | Git 无感管理 | Creator 不知道 Git 是什么，但需要版本管理和安全网 | 已设计 |
| 5 | 依赖与环境自动修复 | npm 报错、版本冲突等 Creator 完全看不懂 | 已设计 |
| 6 | 错误诊断与恢复 | 代码跑不起来，Creator 不知道怎么办 | 已设计 |

> 此列表将持续扩展。识别新守护能力的标准：**如果某个技术操作需要 Creator 理解一个技术概念才能完成，Plugin 就应该把它吸收掉。**

## 能力 1：目录上下文智能引导

> 详细场景与对话示例见 [8 - Install & Init.md → 目录上下文智能引导](8%20-%20Install%20%26%20Init.md#目录上下文智能引导)

### 问题

Creator 不理解"每个 Skill 是独立目录""需要在正确目录打开 Session"等概念。他们会在 Skill A 的目录里说"帮我创建新 Skill"，会在 Workspace 目录里说"开始开发"，会在随机目录里触发 Plugin。

### 处理原则

- Creator 的意图永远合理，Plugin 负责解决"位置不对"
- 能在当前 Session 做的就做（调 API、scaffold 文件），只在必须切 Session 时引导
- 引导切目录时给完整路径，降低操作成本

### 场景清单

| 场景 | Plugin 行为 |
|------|------------|
| 在 Skill A 中创建新 Skill | 在当前 Session 完成创建（API + scaffold），引导开新 Session 打开新目录 |
| 在 Skill A 中切换到 Skill B | 列出项目列表（从 `projects.json`），引导开新 Session |
| 在 Workspace 目录"开始开发" | 列出子项目让选择，或引导创建新项目 |
| 在 Skill 子目录触发 | 向上遍历找到 `.minus.json`，识别项目，提示建议修正工作目录 |
| 在非 Minus 目录 `/minus` 触发 | 列出已有项目，或引导创建 |

## 能力 2：自动环境管理

### 问题

Creator 让 Plugin 修改了代码，但页面没变化。原因可能是：dev server 需要重启、浏览器需要刷新、新依赖需要安装。Creator 不知道这些概念的存在，只会觉得"Plugin 改了但没生效"。

### 处理原则

- **Creator 永远不需要知道"重启"这个概念。** Plugin 改完代码后自动判断、自动执行、自动验证。
- 重启过程对 Creator 透明。如果耗时较长（>3 秒），给一行简短提示。
- 验证改动确实生效后再报告完成。

### 改动类型与处理策略

| 改动类型 | 示例文件 | 预期行为 | Plugin 操作 |
|---------|---------|---------|------------|
| 组件/样式 | `*.tsx`, `*.css`, `*.vue` | HMR 自动热更新 | 验证预览已更新，无需额外操作 |
| 配置文件 | `vite.config.*`, `next.config.*`, `tsconfig.json` | 需要重启 dev server | 自动重启 dev server |
| 依赖变更 | `package.json` | 需要安装 + 重启 | 自动 `npm install` → 重启 dev server |
| 环境变量 | `.env`, `.env.local` | 需要重启 | 自动重启 dev server |
| 静态资源 | `public/*`, `assets/*` | 可能需要刷新 | 自动刷新预览 |
| 后端代码 | `server.*`, `api/*` | 需要重启后端 | 自动重启后端服务 |
| 构建脚本 | `build.*`, `scripts/*` | 无运行时影响 | 不操作 |

### 执行流程

```
Plugin 完成代码修改
    │
    ├── 判断改动类型（基于文件路径和内容）
    │
    ├── 需要安装依赖？
    │   └── 是 → 执行 npm install（静默，失败时进入「依赖自动修复」能力）
    │
    ├── 需要重启服务？
    │   ├── 是，耗时 < 3s → 静默重启
    │   └── 是，耗时 ≥ 3s → 显示「正在重启开发环境...」
    │
    ├── 等待服务就绪
    │   ├── 成功 → 刷新预览
    │   └── 失败 → 进入「错误诊断与恢复」能力
    │
    └── 验证改动生效
        ├── 预览正常 → 报告完成
        └── 预览异常 → 自动排查（进入「错误诊断与恢复」能力）
```

### 对话示例

**正常情况（Creator 无感知）：**

```
Creator > 把标题改成蓝色

Plugin  > ✓ 已将标题颜色改为蓝色。
```

Creator 看到的：标题变蓝了。背后发生了什么：Plugin 改了 CSS，HMR 热更新生效，Plugin 验证预览已更新。Creator 对这一切无感知。

**需要重启（快速）：**

```
Creator > 把端口改成 3001

Plugin  > ✓ 端口已改为 3001，开发环境已重新启动。
```

背后：Plugin 改了配置文件，自动重启 dev server，等待服务在新端口就绪，更新预览地址。

**需要安装依赖：**

```
Creator > 加一个图表功能，用 echarts

Plugin  > 正在安装图表库...
Plugin  > ✓ 图表库已安装，已在页面中添加了示例图表。
```

背后：Plugin 在 `package.json` 加了 echarts，执行 `npm install`，重启 dev server，编写图表组件代码，验证预览。

**重启失败：**

```
Creator > 把数据库连接改成新地址

Plugin  > 已更新数据库连接地址，但重启后端时遇到问题：
          新地址无法连接（连接超时）。
          
          可能的原因：
          · 地址输入有误
          · 数据库服务未启动
          · 网络不可达
          
          要检查一下地址是否正确吗？
```

### 实现要点

**改动类型判断：** 基于文件路径模式匹配。Plugin 维护一个规则表，将文件路径映射到所需操作。规则按项目类型（Vite / Next.js / 纯静态等）加载不同配置。

**服务状态感知：** Plugin 维护当前运行的服务列表（dev server、后端服务等）及其进程 ID，用于判断需要重启哪个服务。

**就绪检测：** 重启后通过轮询健康检查端点（或监听 stdout 的就绪标志）确认服务可用，避免在服务未就绪时刷新预览导致白屏。

**预览验证：** 对于 UI 改动，Plugin 在报告完成前自动检查预览是否正常渲染（无白屏、无报错）。如果预览异常，进入错误诊断流程而非直接报告"已完成"。

## 能力 3：上下文窗口管理

### 问题

Claude Code 的对话有上下文长度限制。当 Creator 在一个 Session 中持续工作，上下文逐渐接近上限时，Claude 的表现会下降——忘记之前说过的话、重复提问、代码生成质量下降。Creator 不知道"上下文窗口"是什么，只会觉得"Plugin 突然变笨了"。

### 处理原则

- Creator 不需要理解"上下文"的概念
- Plugin 在合适的时机主动保存进度并引导开新 Session
- 新 Session 中 Plugin 自动恢复上下文，Creator 感知是"继续工作"而非"重新开始"

### 执行策略

**监测时机：** 每完成一个主要任务后（如完成一个步骤的开发、完成一轮测试），Plugin 评估当前上下文使用量。

**触发阈值：** 当上下文使用超过 ~70% 时，Plugin 在下一个自然断点主动引导切换，而非等到接近 100% 时被动处理。

**切换流程：**

```
Plugin  > 当前 session 内容比较多了，为了保持最佳工作状态，
          建议开一个新 session 继续。

          我已经保存了所有进度：
          · ✓ 数据采集步骤 — 已完成
          · ✓ 竞争度分析步骤 — 开发中，输入定义已完成
          · 待办：竞争度分析的核心逻辑

          请新开一个 session（保持同一个项目文件夹），
          Plugin 会自动恢复到当前进度。
```

**恢复机制：** 新 Session 启动时，Plugin 通过 Memory 读取保存的进度快照，在就绪消息中呈现，让 Creator 感知到连续性：

```
Plugin  > ✓ Minus 已就绪 — 关键词调研 v0.1.0

          上次进度已恢复：
          · ✓ 数据采集步骤 — 已完成
          · → 竞争度分析步骤 — 开发中，输入定义已完成
          
          要继续「竞争度分析」的核心逻辑吗？
```

### 保存内容

| 保存项 | 存储方式 | 用途 |
|--------|---------|------|
| 各步骤开发状态 | 后端 API | 跨 Session 恢复进度 |
| 当前工作上下文 | Claude Code Memory | 恢复对话连续性（正在做什么、做到哪里、遇到什么问题） |
| 未完成的决策 | Claude Code Memory | 避免重复讨论已经做过的决定 |

## 能力 4：Git 无感管理

### 问题

Creator 不知道 Git 是什么，但需要版本管理提供的安全网——能回退到之前的状态、能记录每个阶段的成果。如果 Plugin 需要 Creator 理解 commit、branch、merge 等概念，大部分 Creator 会直接放弃。

### 处理原则

- Creator 不需要知道 Git 的存在
- Plugin 在关键节点自动保存版本，形成安全网
- 出问题时 Plugin 用业务语言描述回退选项，而非 Git 术语

### 自动保存时机

| 触发事件 | 保存行为 | 对 Creator 说什么 |
|---------|---------|-----------------|
| 完成一个步骤的开发 | `git commit -m "完成「数据采集」步骤"` | 不说，静默 |
| 发布前 | `git tag v0.1.0` | 不说，静默 |
| 开始大改动前 | `git commit` 当前状态 | 不说，静默 |
| Creator 要求"撤回刚才的修改" | `git checkout` 相关文件 | "已撤回，恢复到修改前的状态" |
| Creator 要求"回到上一个版本" | `git revert` 或 `checkout` 到上一个节点 | "已恢复到「数据采集」完成时的状态" |

### 对话示例

**Creator 想撤回修改：**

```
Creator > 刚才改的不好，能恢复吗？

Plugin  > 已恢复到修改前的状态。刚才的改动是：
          · 修改了标题样式
          · 调整了布局间距

          这些改动已撤回，页面已恢复。
```

**Creator 想回到更早的状态：**

```
Creator > 我记得之前有个版本挺好的，能回去吗？

Plugin  > 你的项目有这些保存点：
          
          1. 5 分钟前 — 修改了输出展示样式
          2. 2 小时前 — 完成「竞争度分析」步骤
          3. 昨天 — 完成「数据采集」步骤
          
          要恢复到哪个？

Creator > 2

Plugin  > ✓ 已恢复到「竞争度分析」步骤完成时的状态。
          之后的样式修改已撤回。
```

注意：Plugin 用"保存点"而非"commit"，用"恢复"而非"checkout"或"revert"。

### 实现要点

**Commit message 规范：** 使用业务语言而非技术语言。格式为 `[步骤名] 动作描述`，例如 `[数据采集] 完成输入定义` 。这样在列出保存点时可以直接展示给 Creator。

**分支策略：** 不使用分支。单个 Creator 开发单个 Skill，线性历史足够。如果未来支持多人协作再引入分支概念。

**`.gitignore` 预配置：** Scaffold 时生成完善的 `.gitignore`，确保 `node_modules/`、`.env`、密钥文件等不被提交，Creator 不需要知道忽略规则。

## 能力 5：依赖与环境自动修复

### 问题

`npm install` 失败、版本冲突、Node.js 版本不兼容——这些错误信息对 Creator 来说是天书。他们无法从 `ERESOLVE unable to resolve dependency tree` 中得到任何有用信息。

### 处理原则

- 依赖安装失败时，Plugin 先自动尝试修复，而非把错误甩给 Creator
- 自动修复失败后，用通俗语言说明问题和解决方案
- 绝不展示原始 npm 错误日志（除非 Creator 主动要求看）

### 自动修复策略

| 错误类型 | 自动修复方式 | 修复失败时告知 Creator |
|---------|------------|---------------------|
| 版本冲突（ERESOLVE） | 尝试 `--legacy-peer-deps`，或调整冲突依赖版本 | "有两个组件库版本不兼容，我尝试了自动修复但没成功。需要调整方案。" |
| 网络超时 | 自动重试（最多 3 次），尝试切换 registry | "网络连接不稳定，暂时无法安装。请检查网络后告诉我重试。" |
| Node.js 版本不兼容 | 提示需要的版本并给出升级指引 | "当前环境需要升级，我来帮你操作。"（若 Plugin 有权限则自动处理） |
| 磁盘空间不足 | 清理 npm cache | "存储空间不足，我已清理了缓存。如果还不够，可能需要清理一些文件。" |
| 权限错误 | 修正目录权限 | "文件权限有问题，我来修复。" |
| 包不存在（404） | 检查包名拼写，搜索相似包 | "找不到这个包，你是不是要装 xxx？" |

### 对话示例

**自动修复成功（Creator 无感知）：**

```
Creator > 加个日期选择器

Plugin  > ✓ 已添加日期选择器组件。
```

背后：Plugin 安装了 date-picker 库，遇到版本冲突，自动用 `--legacy-peer-deps` 解决，重启 dev server，一切正常。Creator 对中间过程无感知。

**自动修复失败：**

```
Creator > 加个 3D 图表

Plugin  > 想添加 3D 图表功能，但遇到了一个兼容性问题：
          3D 图表库需要的图形引擎版本和当前项目不兼容。
          
          有两个方案：
          1. 用另一个 3D 图表库（xxx），兼容当前项目
          2. 升级图形引擎，但可能影响已有的 2D 图表
          
          你倾向哪个？
```

## 能力 6：错误诊断与恢复

### 问题

代码运行出错时，Creator 面对的是浏览器白屏、控制台红色错误、或服务启动失败的日志。他们无法从这些信息中判断问题出在哪里，更不知道如何修复。

### 处理原则

- Plugin 主动检测错误，而非等 Creator 发现并报告
- 能自动修复的直接修，修完告知 Creator 发生了什么
- 不能自动修复的，用通俗语言解释并给出选项
- 绝不把原始错误堆栈直接丢给 Creator

### 错误检测机制

| 检测方式 | 触发条件 | 覆盖场景 |
|---------|---------|---------|
| 预览监控 | 代码修改后检查预览是否正常渲染 | 白屏、渲染错误、组件崩溃 |
| 控制台监听 | 检测浏览器/Node.js 控制台的 error 级别输出 | 运行时异常、未捕获错误 |
| 服务健康检查 | dev server / 后端服务的心跳检测 | 服务崩溃、端口冲突 |
| 构建监控 | 编译/构建过程的退出码和输出 | 语法错误、类型错误、导入错误 |

### 自动修复能力

| 错误类型 | 自动修复方式 | 示例 |
|---------|------------|------|
| 语法错误 | 分析错误位置，自动修正 | 缺少括号、多余逗号 |
| 导入错误 | 修正导入路径或添加缺失导入 | `Module not found` |
| 类型错误 | 自动添加类型转换或修正类型 | TypeScript 类型不匹配 |
| 端口冲突 | 自动切换到可用端口 | `EADDRINUSE` |
| 组件渲染崩溃 | 分析错误边界日志，修复组件 | React Error Boundary 触发 |

### 对话示例

**自动修复（Creator 无感知）：**

Plugin 改了一段代码，预览出现白屏。Plugin 检测到是导入路径错误，自动修正，预览恢复。Creator 看到的只是正常的完成消息，不知道中间出过错。

**自动修复 + 简短告知：**

```
Plugin  > ✓ 图表组件已添加。
          （修复了一个小问题：组件间的数据格式不匹配，已自动调整）
```

**无法自动修复：**

```
Plugin  > 添加图表组件后页面出现了问题。
          
          原因：图表需要的数据格式和当前接口返回的不一致。
          接口返回的是 [{name, value}] 格式，但图表需要 {x: [], y: []} 格式。
          
          我可以：
          1. 加一个数据转换层，把接口数据转成图表需要的格式
          2. 换一个支持当前数据格式的图表组件
          
          选哪个？
```

### 恢复兜底

当自动修复失败且 Plugin 无法确定修复方案时，利用 Git 无感管理（能力 4）提供安全网：

```
Plugin  > 这个改动引起了一些问题，我尝试了几种修复方案都不理想。
          
          建议先恢复到改动前的状态，然后我们换个思路来实现。
          要恢复吗？
```

这让 Creator 永远不会陷入"代码坏了不知道怎么办"的困境。

## 守护能力扩展指南

### 如何识别新的守护能力

当满足以下任一条件时，应新增守护能力：

1. **Creator 需要理解一个技术概念才能继续工作** — 例如"什么是环境变量""为什么要重启"
2. **某个操作经常导致 Creator 卡住或求助** — 从实际使用数据中发现的高频问题
3. **错误信息对 Creator 无意义** — 原始错误信息无法指导非技术用户行动
4. **Plugin 可以自动完成，但当前需要 Creator 手动操作** — 减少不必要的手动步骤

### 守护能力的设计模板

每项新能力应包含：

```
## 能力 N：[能力名称]

### 问题
Creator 遇到了什么？为什么他们无法自行解决？

### 处理原则
Plugin 怎么处理？（3-5 条）

### 场景与对话
每个典型场景的处理方式和示例对话。
- 自动修复成功（Creator 无感知）
- 自动修复成功 + 简短告知
- 无法自动修复 → 用通俗语言给出选项

### 实现要点
技术层面的关键决策和注意事项。
```

### 跨能力协作

守护能力之间不是孤立的，它们形成协作网络：

```
代码修改
  → 自动环境管理（判断是否需要重启）
    → 重启失败 → 错误诊断与恢复（排查原因）
      → 需要装新依赖 → 依赖自动修复（处理安装问题）
        → 安装失败不可修复 → Git 无感管理（回退到安全状态）
          → 切换 Session → 上下文窗口管理（保存进度到新 Session）
```

每个环节失败时自动流转到下一个能力，形成完整的安全网。Creator 始终在 Plugin 的守护范围内。
