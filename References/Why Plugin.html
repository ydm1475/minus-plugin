MINUS CREATOR PLUGIN

Why Plugin?

不自建开发者平台，不用单一 Skill，而是选择 Plugin 作为 Minus 创作者开发模式的核心架构决策

Decision 1

为什么不自建开发者平台

面对"如何让创作者开发 Skill"这个问题，第一个被排除的方案是自建 Web 平台或桌面软件。

01

本质上是在复制 AI 编程环境

创作者开发 Skill 需要的核心能力——LLM 对话、代码生成、调试运行、上下文管理——就是 Claude Code 已经具备的全部能力。自建平台意味着通过接入大模型 API 把这些能力再实现一遍，同时还要自己解决 session 管理、context 压缩、Memory 持久化等工程难题。

02

维护成本远超收益

Claude Code 每次迭代都会带来新能力（更强的上下文管理、更好的工具调用、更精准的代码生成）。自建平台需要持续跟进这些更新，否则就会落后于原生开发体验。团队资源应该聚焦在 Minus 的核心价值——数据与分析能力，而不是维护一个 IDE。

03

创作者已经在用 Claude Code

Minus 的目标创作者群体是 AI 工具的重度用户。他们已经熟悉 Claude Code 的交互模式。让他们切换到一个全新的开发环境，不仅增加学习成本，还在体验上做了降级——没有哪个自建平台能在短期内比 Claude Code 做得更好。

04

上线速度差异巨大

自建平台从前端、后端、LLM 集成、测试环境到上线，需要数月工程投入。而基于 Claude Code 的方案，核心开发工具链已经就绪，Minus 只需定义规范和发布通道，几周内即可让创作者开始开发。

✓ 决策结论

不自建开发者平台。Minus 的创作者开发环境直接运行在 Claude Code 中。Minus 提供的是规范、API 和发布通道，而不是另一个 IDE。

Decision 2

为什么选择 Plugin 模式

确定了基于 Claude Code 之后，接下来要选择具体的扩展机制。Claude Code 的 Plugin 是一个组合系统，而不是单一的扩展点。

Agent

✓ 核心优势

Plugin 是一个系统，不是一个文件。它让 Minus 可以为开发流程的每个环节选择最合适的机制——Skill 负责引导对话，Hook 负责自动化，MCP 负责外部集成，Agent 负责复杂编排。这种组合式架构确保了体验的完整性和可扩展性。

Decision 3

为什么不选 Skill 模式

在选择 Plugin 之前，最初的方案是用一个单独的 Skill（即一个 .md 文件）来承载整个创作者开发流程。随着架构深入，这个方案暴露了根本性的限制。

✗ 没有生命周期管理

Skill 是被动触发的——只有用户明确调用时才激活。无法实现"进入项目目录自动加载环境"这种 SessionStart 行为，也无法在工具调用后自动保存进度。

需要 Hook

✗ 无法集成外部服务

Skill 本身不能启动 MCP Server。这意味着 API 发现、OAuth 认证流程、发布上传等需要与 Minus 后端交互的操作，纯 Skill 模式下无法实现。

需要 MCP Server

✗ 单文件无法承载复杂结构

一个 Skill 是一个 .md 文件。当系统需要管理多个子 Skill（/minus、/minus-publish）、配置文件、模板库、工具函数时，单文件模式在组织和维护上都不可行。

需要 Plugin 目录结构

✗ 缺乏认证模型

Skill 没有内置的凭证管理机制。创作者的 Minus 平台登录凭证、Skill 的运行时 API Key，都需要一个结构化的存储和读取方案，这超出了单个 Skill 文件的能力。

需要 Plugin 状态管理

✗ 不支持上下文管理策略

长对话中的上下文窗口耗尽是真实痛点。Skill 无法注册 PostToolUse Hook 来自动检测上下文使用率，也无法在接近极限时主动保存进度并引导创作者开启新 Session。

需要 Hook + Memory

✗ 产物分发受限

Skill 通过 description 字段的自然语言匹配来触发，适合轻量工具。但 Minus Creator Plugin 需要完整的安装流程（clone 目录 → 注册权限 → 配置凭证），这需要结构化的 manifest 和安装脚本。

需要 plugin.json + 安装流程

✓ 根本区别

Skill 是一条指令，Plugin 是一个系统。Skill 适合定义"Claude 应该怎么做某件事"，它本质上是一段结构化的 prompt。而 Minus 需要的是一个完整的开发环境——有启动流程、有状态管理、有外部集成、有自动化行为。这不是一个 prompt 能承载的复杂度。

最终选择：Plugin

自建平台是重复建设——用数月时间复制 Claude Code 已有的能力，还要持续维护。排除。

Skill 模式能力不足——它是一条高质量的 prompt，适合轻量级任务引导，但无法支撑完整的开发环境（无生命周期、无外部集成、无结构化状态管理）。排除。

Plugin 模式恰好在正确的抽象层级——不重复建设，直接站在 Claude Code 上；同时通过 Skills + Hooks + MCP + Agents 的组合，获得了 Skill 单文件模式缺失的全部能力。它是"不自建平台"和"提供完整开发体验"之间的精确平衡点。
