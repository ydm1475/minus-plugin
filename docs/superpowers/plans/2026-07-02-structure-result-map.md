# minus-structure 交互改进方案：地图、可见反馈与措辞理由

> 依据：诺曼视角审查 minus-structure 的四点结论——
> ① minus-step 已有「开场报总数、追问带锚点、切换报站」（见
> [2026-07-02-node-dev-map.md](2026-07-02-node-dev-map.md)），但同一产品的另外两段旅程——
> 结构设计阶段（用户需要提供什么信息 + 流程分几步）、跨步骤的整体进度、结果设计阶段
> （结果摘要 + 下载内容）——都没有地图，Creator 被逐段扔进黑箱，靠反复追问兜底而非靠地图预判；
> ② 输入区域代码改完后是一段静默的黑箱，没有任何一句话邀请 Creator 去看，评估沟壑；
> ③ 数量三选项菜单里"至少两个"是工程视角的区间枚举，用户不一定知道为什么会有这个选项。
> 原则：与 node-dev-map 同一套语法（地图报"城市"不报"转弯"；单源定义；不做的事同样排除），
> 不重新发明一套新话术体系，保证跨 skill 一致。

**贯穿全文的示例场景**：Creator 在做一个「亚马逊关键词分析」Skill，共 4 步
（① 选类目 → ② 关键词筛选 → ③ 竞品分析 → ④ 生成报告）。
下面各改动按 Creator 实际经历的时间顺序排列：先结构设计，再跨天开发步骤，最后结果设计。

---

## 改动 1：structure-design.md 顶层地图（任务清单单源 + 开场报总数 + 区块切换报站）

**位置**：`skills/minus-structure/structure-design.md` 顶部，「## 第一步：确定输入」之前。

**内容**：新增任务清单单源表，仅适用于「初次设计」路径（SKILL.md 路由表里"已有步骤代码？否"那一支；
结构变更路径见「不做的事」）：

```markdown
### 任务清单（单源定义，仅适用于初次设计）

本清单是初次结构设计两件事的唯一权威定义，开场白、切换报站话术都从这里取名字。

| 序号 | 名称 |
|------|------|
| ① | 用户需要提供什么信息 |
| ② | 流程分几步以及每步具体做什么 |
```

「首次提问必须原样输出」的开场白前追加一句报总数（原有开场白内容不变，只是前面加一句）：

```
我们要确认两件事：用户需要提供什么信息、流程分几步以及每步具体做什么——先看第一件。
```

第一步「✓ 输入定义确认」之后、进入「第二步：拆解步骤」之前，加一句切换报站（半句收束 + 半句开新）：

```
第一件事确认完，接下来看流程分几步以及每步具体做什么。
```

**设计说明**：
- 任务清单命名沿用 node-dev-map 的检验标准——名称要让 Creator 听到后的预期和实际被问的问题一致，且优先直接回声接下来会问的原话（零翻译成本），而不是用"输入定义""步骤拆解"这类工程词或缩写。"输入定义"里的"定义"暗示要写规范/schema，实际发生的事只是说清楚用户要交什么；"步骤拆解"里的"拆解"本义是"把已完整的东西拆开"，但初次设计时 Creator 面前没有已存在的结构可拆，用词和场景错位（该词真正适配的场景是「已有项目的步骤结构变更」）。
- ②保留"每步做什么"而不简化成"分几步"——原话本来就是两问一体（"Skill 要分几步完成？每一步做什么？"），标签若只回声"分几步"，Creator 可能以为这一步只需回答一个数字，答完就结束，漏答"每步做什么"那半截。
- 只有两件事，仍然值得报总数——地图的价值不在于"件数多才需要"，而在于"不报，Creator 就不知道自己在哪站"。两件事和四件事一样会让人在第一件事问完后困惑"结束了吗"。
- ①②标签较长（不同于 node-dev.md 里"数据需求""数据处理"等 4 字标签），是直接回声保真度换来的代价，权衡后优先保真度——node-dev.md 的"怎么交给下一步"同样是完整短句而非纯名词，先例已有。

**场景示例——改动前 vs 改动后**：

> **改动前**（Creator 说"我想做一个亚马逊关键词分析 Skill"后，Agent 直接开问）：
>
> > Agent：用户使用这个 Skill 时，需要提供什么信息？
> > 比如关键词、ASIN、品类……不限于这些，按你的思路说就行
>
> Creator 心里：不知道回答完这个还有几轮，也不知道回答完是不是就能看到 Skill 雏形了。
>
> **改动后**（先给地图，再开问）：
>
> > Agent：我们要确认两件事：用户需要提供什么信息、流程分几步以及每步具体做什么——先看第一件。
> > 用户使用这个 Skill 时，需要提供什么信息？
> > 比如关键词、ASIN、品类……不限于这些，按你的思路说就行
>
> Creator 答完提示语，看到"✓ 输入定义确认"后：
>
> > Agent：第一件事确认完，接下来看流程分几步以及每步具体做什么。
> > 下一个问题：拿到用户的关键词后，Skill 要分几步完成？每一步做什么？按你的思路说就行。（后续随时都可以调整）
>
> Creator 清楚知道自己走完了第一件事、正在进入第二件也是最后一件，而不是猜"还要问多久"。

## 改动 2：node-dev.md 开场白追加总步数

**位置**：`skills/minus-step/node-dev.md` 第 56 行「进场开场白」模板。

**改动前**：

```
「开发第 N 步[步骤名]（最后一步时在步骤名后加"（也是最后一步）"）。我们要确认 {件数} 件事：{按序号罗列适用区块名}——先看第一件。」
```

**改动后**：

```
「开发第 N 步[步骤名]（共 M 步{最后一步时追加"，这是最后一步"}）。我们要确认 {件数} 件事：{按序号罗列适用区块名}——先看第一件。」
```

M 直接复用第 52 行已经读取的 `.minus/total-steps`，不新增读文件。

**设计说明**：
- 结构设计阶段确认"共 4 步"之后，Creator 可能隔天甚至隔周才回来开发第 3 步——中间那次对话里说过的总数早就忘了。`node-dev.md` 每次开发一步都会重新读 `total-steps`（用于判断是否最后一步），只是没把这个数字讲给 Creator，等于系统知道路况却不告诉司机。
- **"第 N 步"和"[步骤名]"必须挨在一起，中间不能插别的信息**——编号和名字本来就是"这是第几步、叫什么"同一件事的两面，插进去会把读者的注意力从"哦这是竞品分析那一步"打断成"共 4 步……哦对了竞品分析"。第一版方案把"（共 M 步）"插在两者中间，犯了这个错；改法是把"共 M 步"挪到步骤名之后，和"这是最后一步"合并进同一个括号，两条补充信息一起说，不再打断编号和名字的连续性。
- 括号内"共 M 步"和"这是最后一步"合并成一句，用逗号分隔，不是两层嵌套括号——避免括号套括号的阅读负担。

**场景示例**：

> Creator 三天前拆完步骤，今天回来说"继续开发第 3 步"。
>
> **改动前**：
>
> > Agent：开发第 3 步「竞品分析」。我们要确认四件事：数据需求、数据处理、展示内容、怎么交给下一步——先看第一件。
>
> Creator 心里：这是第 3 步，那……一共几步来着？还剩几步没做完？
>
> **改动后（非最后一步）**：
>
> > Agent：开发第 3 步「竞品分析」（共 4 步）。我们要确认四件事：数据需求、数据处理、展示内容、怎么交给下一步——先看第一件。
>
> 不用回忆，一眼知道还剩 1 步。
>
> **改动后（最后一步，两条补充信息合并进同一括号）**：
>
> > Agent：开发第 4 步「生成报告」（共 4 步，这是最后一步）。我们要确认三件事：数据需求、数据处理、展示内容——先看第一件。

## 改动 3：结果设计阶段报站

**位置**：`skills/minus-structure/scripts/generate-result-design.sh`——门禁通过后的引导输出部分（`GATE_PASSED` 之后的「两维度引导」文本），以及 `confirm summary` 分支里转达给 Creator 的话术。

**改动前**（「示例提问（原样输出，不要改写）」段落）：

```
「所有步骤开发完成。各步骤产出的数据：」
「 · 步骤 1：{步骤名} — {通俗语言描述}」
「 · 步骤 2：...」
「」
「Skill 运行结束后，结果页底部会有一段摘要来总结分析结论。」
「这段摘要由大模型在运行时基于实际数据自动生成，还是你来定义模板？」
```

**改动后**（报总数插在"看完产出数据"和"问第一件事"之间，不堵在最前面）：

```
「所有步骤开发完成。各步骤产出的数据：」
「 · 步骤 1：{步骤名} — {通俗语言描述}」
「 · 步骤 2：...」
「」
「进入结果呈现设计。我们要确认两件事：结果摘要、下载内容——先看结果摘要。」
「Skill 运行结束后，结果页底部会有一段摘要来总结分析结论。」
「这段摘要由大模型在运行时基于实际数据自动生成，还是你来定义模板？」
```

`confirm summary` 分支（脚本第 33-36 行）转达给 Creator 的下载内容提问前，加一句切换报站：

```
echo "「结果摘要确认好了，下一件事——」"
echo "「用户可以下载哪些内容？比如 Excel、HTML 报告……」"
echo "「你想提供什么下载格式？」"
```

**设计说明**：
- **报总数不能堵在数据列表前面。** 第一版方案把"进入结果呈现设计，两件事……"塞进了最开头，Creator 还没看到自己四步都产出了什么，就先被拽去关心"接下来要确认哪两件事"——先给路标、后给内容，本末倒置。改法是让 Creator 先看到具体、踏实的东西（自己四步的产出），看完再切换到抽象的"接下来要做什么"，顺序对应认知负荷从低到高，不是反过来。
- 这是三段旅程里唯一一段完全由脚本（而非 .md 话术）驱动引导文本的地方，报总数这句必须加进脚本的 heredoc 里、且插在正确位置，而不是寄望 Agent 在读脚本输出后自己脑补加一句或自己决定插在哪——这正是「结构设计文档」审查里第一层问题的教训：不能靠 Agent"自觉"补一句话或自觉排序，得让它成为脚本原样转达文本、且顺序固定的一部分。
- `tests/shell-scripts.test.sh:1985` 断言 OUTPUT 同时包含"结果摘要"和"下载内容"（子串匹配），`:1998` 断言精确子串"只有 Creator 明确确认后，才能继续进入下载内容"——两处都不受本次改动影响，新增文本是插入不改变既有子串。

**场景示例**：

> 4 步全部开发完成，Creator 收到测试邀请并确认整体测试通过后：
>
> **改动前**：
>
> > Agent：所有步骤开发完成。各步骤产出的数据：
> > · 步骤 1：选类目 — 品类列表及热度
> > ...
> > Skill 运行结束后，结果页底部会有一段摘要来总结分析结论。这段摘要由大模型在运行时基于实际数据自动生成，还是你来定义模板？
>
> Creator 心里：问完这个是不是就结束了？还有别的要问吗？
>
> **改动后**：
>
> > Agent：所有步骤开发完成。各步骤产出的数据：
> > · 步骤 1：选类目 — 品类列表及热度
> > ...
> >
> > 进入结果呈现设计。我们要确认两件事：结果摘要、下载内容——先看结果摘要。
> > Skill 运行结束后，结果页底部会有一段摘要来总结分析结论。这段摘要由大模型在运行时基于实际数据自动生成，还是你来定义模板？
>
> Creator 先看清自己四步做出了什么，再知道接下来要确认两件事、现在是第一件——两件事都有着落，顺序也不别扭。
>
> 摘要确认完成后：
>
> > Agent：结果摘要确认好了，下一件事——用户可以下载哪些内容？比如 Excel、HTML 报告……你想提供什么下载格式？

## 改动 4：代码改完后的可见反馈（结束评估沟壑）

**位置**：`skills/minus-structure/structure-design.md`「确认后更新前端代码」的 9 步 checklist 之后
（含"如果 Home 已有输入组件（切换类型场景）"分支之后），「⛔ 禁止：」几条规则之前，新增一条必须原样输出的收尾话术。

**内容**：

```
这部分已经改好了，刷新页面看一下效果——提示语和交互方式是你想要的样子吗？
```

标记为原样输出、不改写，与本文件里其他强制话术（如「开场白」）同一约束等级。

**设计说明**：
- 现状：Creator 说完 placeholder，得到一句"✓ 输入定义确认"，Agent 随即改 9 处前端代码，改完转身直接问步骤拆解——中间没有任何一句话告诉 Creator"改完了、可以去看"。执行完成、结果不可见，是典型的评估沟壑：恒温器最惹人烦的不是调温慢，是调完之后没有任何迹象表明它听见了。
- **不复用 `update-progress.sh` 里的"测试邀请"话术**：那段话术假设整条 pipeline 已经可以跑通（"把整个流程从头到尾跑一遍"），但这个节点上步骤代码还是 `# TODO` 骨架，根本跑不起来，Creator 只能看到输入区域本身。硬套现成话术等于承诺一个此刻做不到的动作（走完整流程），会制造新的"地图撒谎"，比不说还糟。
- 两处话术的**结构**是一致的——"代码改完 → 必须有一句固定话术邀请 Creator 去看"，只是范围不同（一个是整条 pipeline，一个是刚改完的这部分输入区域）。这条规则先在 `.md` 里落地；如果未来这段代码改动也想改成脚本驱动（类似 `update-progress step-done` 那样有独立的进度落盘），可以直接复用这个模式，不用重新设计。

**场景示例**：

> Creator 确认完提示语"输入主关键词，如 wireless earbuds"后：
>
> **改动前**：
>
> > Agent：✓ 输入定义确认。
> > 输入定义确认完，接下来看流程分几步以及每步具体做什么。
> > 下一个问题：拿到用户的关键词后，Skill 要分几步完成？……
>
> Creator 心里：这部分到底改成什么样了？我完全没看到。
>
> **改动后**：
>
> > Agent：✓ 输入定义确认。
> > 这部分已经改好了，刷新页面看一下效果——提示语和交互方式是你想要的样子吗？
>
> Creator 确认没问题后：
>
> > Agent：输入定义确认完，接下来看流程分几步以及每步具体做什么。
> > 下一个问题：……

## 改动 5：数量三选项按情况内嵌理由

**位置**：`skills/minus-structure/structure-design.md` 第 22-27 行，数量追问模板（"用户可以输入几个？只能一个、一个或多个、还是至少两个"）之后。

**内容**：三个选项本身保留，追加一条规则——理由能确定就直接写进选项里问出来，不确定就先不带、等 Creator 犹豫或反问再补：

```markdown
⛔ 三选项菜单本身固定不变。"至少两个"这个选项能不能直接带理由，取决于此刻能不能从 Creator
已经说过的内容里确定一个站得住脚的理由：
- 能确定（如已经知道这是"竞品分析"这类对比型 Skill）→ 直接把理由写进选项一起问出来，
  不用等 Creator 犹豫或反问，减少一次来回。格式：「……还是至少两个（比如做对比分析，通常
  至少要两个）」，括号内容按当前 Skill 动态生成，不要照抄本例。
- 不能确定（还不清楚 Skill 具体用途）→ 先原样输出三选项，不编理由；等 Creator 犹豫或反问
  "至少两个是什么意思"时再结合已知信息补一句。
⛔ 两种情况下理由都必须结合 Creator 已经说过的 Skill 用途动态生成，禁止仅凭输入类型本身
臆测（同「语境复述」同一原则）——编一个不确定的理由比不给理由更误导。
```

**设计说明**：
- 原方案（"只在 Creator 犹豫或反问时才补理由"）靠 Agent 自行判断"这算不算犹豫"，是一个软触发，判断不准就会漏补——违反 CLAUDE.md 第一条原则"能硬编码的别靠 Agent 自觉"。改成"能确定就直接内嵌"，把判断的分支从"要不要补"变成"能不能确定"，后者有更明确的依据（此刻是否已知 Skill 用途），不依赖对 Creator 语气的解读。
- 但不能变成"永远内嵌"：追问数量这一步经常发生在 Creator 只说了一个类型词（如示例 B 里只回"关键词"）、还没走到「语境复述」分支的时刻，此时 Agent 大概率还不知道 Skill 具体用途。这时候被强制要求带理由，会逼着编一个不确定的理由，直接撞上改动 1 里"禁止仅凭输入类型本身臆测业务场景"的规则。所以保留"不能确定就不编"这条退路，两条规则不冲突。
- "只能一个、一个或多个、还是至少两个"里，"至少两个"是工程视角的区间枚举（对应校验函数的 `min=2`），不是用户脑子里的自然分类。选项本身没错，但对着一个抽象数字选，用户不一定知道"至少两个"在什么场景下才是对的答案——这是理由存在的价值所在，不是可有可无的装饰。
- 理由不能通用一句话打天下——"至少两个"在对比分析类 Skill 里的理由（"要对比就得有两个对象"）和在批量处理类 Skill 里的理由（"批量意味着不止一个"）并不相同，套用错的理由比不给理由更误导，所以两种路径都要求动态生成。

**场景示例**：

> Creator 说要开发"竞品分析"Skill，通过语境复述已确认输入类型是 ASIN、用途是互相比较。
>
> **改动前**：
>
> > Agent：用户可以输入几个？只能一个、一个或多个、还是至少两个
> > Creator：至少两个是什么意思？
> > Agent：就是最少要输入两个 ASIN。
>
> Creator 心里：为什么会有"至少两个"这个选项，我为什么要选它？
>
