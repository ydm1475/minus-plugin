## 定位

本 skill 只服务插件开发者自己（不会打包进发给 Creator 的插件——`pack.sh` 只收
`plugins/claude/minus-creator/`，不动仓库根目录的 `.claude/`）。因此不做 cwd
判断：只要这个 skill 能被加载，就已经说明当前是插件开发者在工作，不需要再
像 `minus-diagnose` 那样先确认"是不是在做 Minus 开发"。

`minus-diagnose` 是给 Creator 用的，负责诊断 Creator 自己项目里的故障。本
skill 负责一个更早的判断：**这个症状的根子，到底在插件源码里，还是只是某
个项目自己的情况**——如果误判成"只是项目个例"直接局部打补丁，插件里的通
病会一直留着，下一个 Creator 还会踩到同样的坑。

## 1. 触发信号

判断依据是**内容**，不是 cwd 或关键词：用户是否贴出/引用了具体执行记录
（日志片段、报错堆栈、截图描述的具体现象）。纯设计讨论、需求讨论不触发本
流程。

## 2. 标注检查

用户是否已经明确标注归属（例如"这是插件通病"或"这只是那个项目的个
例"）：

- 已标注 → 直接采信，跳到第 4 步按标注结果路由
- 未标注 → 进入第 3 步强制反查

## 3. 强制反查（无标注时）

从症状出发，定位到对应的插件源码：skill.md 模板、生成脚本、
`mcp-servers/minus-platform/` 逻辑等。判断这段逻辑是：

- **通用生成路径**（所有 Creator 项目都会走到这段逻辑）——静态比对症状能
  否在这段源码里重现，能重现即可判定
- **项目定制内容**（这个项目自己的业务代码/数据，插件只是生成了脚手架，
  之后的内容是 Creator 自己写的）——判定为项目个例

静态比对给不出结论时，才升级到实测：起一个测试项目复现，验证同样的路径
是否触发同样症状。实测是兜底，不是默认路径。

## 4. 输出判定结果并路由

- **插件通病** → 给出插件源码的文件路径 + 行号，说明问题出在生成逻辑的
  哪一步，进入正常的插件修复流程（改代码、提交前跑 `tests/run-all.sh`）
- **项目个例** → 明确告知"这不是插件的问题，不该改插件"，到此为止。本
  skill 不自动去修那个项目——分诊和修复是两件事，职责不焊在一起
