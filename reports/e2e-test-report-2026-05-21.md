## 一、用户反馈的问题

### BUG-1: 端口占用处理不当

**现象**: 端口 5173 被另一个项目 (create_skill) 的 Vite 占用，Agent 检测到端口被占后"跳过启动"，但打开了 `http://localhost:5173`，实际展示的是别的项目。用户手动报告后 Agent 才处理。

**根因**: SKILL.md 第 132-133 行的逻辑有缺陷：
```
lsof -i :{port} → 端口已被占用 → 跳过启动
```
只检查端口是否被占，不检查**是谁占的**。如果是别的项目占了，当前项目的 dev server 根本没启动。

**控制源**: SKILL.md 的端口检查逻辑

**修复方案**: 端口检查后，必须验证占用进程是否属于当前项目：
```bash
lsof -i :{port} -t | xargs -I{} ps -p {} -o command= | grep -q "$(pwd)"
```
- 是当前项目 → 跳过启动
- 不是当前项目 → 用 port-detector.sh 找可用端口，或提示用户

### BUG-2: 维度③提问格式不清晰（选项没换行）

**现象**: 截图 1 中，维度③的问题这样显示：
> 处理逻辑已写好。下一个问题：这一步要展示什么给用户看？比如一个数据表格、一段文字摘要、一个评分卡片……还有，需要传什么数据给下一步？

选项挤在一起，用户难以区分。

**根因**: node-dev.md 第 61-63 行定义了三行独立输出：
```
「处理逻辑已写好。下一个问题：这一步要展示什么给用户看？」
「比如一个数据表格、一段文字摘要、一个评分卡片……」
「还有，需要传什么数据给下一步？」
```
但 Agent 实际输出时把三行合成了一段。说明 Agent 没有严格遵守"原样输出"指令。

**控制源**: node-dev.md 维度③的输出指令

**修复方案**: 加强格式约束，每个子问题用换行分隔，且加编号：
```
「处理逻辑已写好。下一个问题：」
「这一步要展示什么给用户看？」
「  比如一个数据表格、一段文字摘要、一个评分卡片……」
「还有，需要传什么数据给下一步？」
```
同时在所有维度的提问模板统一加上空行分隔，确保 Agent 逐行输出。

### BUG-4: 用户确认后刷新页面数据丢失

**现象**: 第 1 步用户点击"确认并继续"后，刷新浏览器，已确认的数据消失。

**根因分析**: 框架的 defineWidgetStep 支持数据持久化，确认后 userInput 会存到后端 NodeShape.userInput。但可能存在两种情况：
1. 后端 `input_required` 状态下，用户确认后 payload 没有正确写入 userInput
2. 前端刷新后没有从 stepOutputs 恢复 userInput

**控制源**: widget-framework 的 defineWidgetStep + 后端 SDK 的 StepOutcome.input_required 持久化逻辑

**修复方案**: 需要在 SDK 和 widget-framework 两层验证：
1. 后端确认 `onResolve` 回调是否将 userInput 写入持久存储
2. 前端确认刷新后是否从 stepOutputs 恢复 userInput[confirmedKey]
3. 写 E2E 测试覆盖"确认→刷新→数据恢复"路径

### BUG-5: 第 2 步表格未使用 SelectableTable readonly 模式 + 摘要行未展示

**现象**: 
- 第 2 步（热销 ASIN）用的是手写 HTML table 而非 SelectableTableWidget 的 readonly 模式
- 刷新后"共找到 N 个热销 ASIN"的摘要行没有显示

**根因**: Agent 在第 2 步（最后一步，不需要用户确认）写了自定义 HotAsinTable 组件，手动拼接 `<table>` HTML。但框架已有 SelectableTableWidget 的 Readonly 组件，可以直接用。

手写组件的问题：
1. 不走框架的渲染统一逻辑，样式和交互不一致
2. 刷新后 data 可能还没到（异步加载），摘要文字依赖 data.total，为 undefined 时不渲染
3. 后续维护成本高，每个 Skill 都手写一遍

**控制源**: node-dev.md 维度③的代码编写规范

**修复方案**:
1. node-dev.md 维度③增加规范：纯展示表格统一用 `defineWidgetStep` + `SelectableTableWidget`，只是不传 `interactiveProps`（readonly 模式）
2. 框架层面：SelectableTableWidget.Readonly 在 data 尚未就绪时应展示 loading 状态，而非空白

### BUG-6: 用户确认的交互方式应默认用弹框（dialog）

**现象**: 截图 2 显示，第 1 步的 SelectableTable 是内联（inline）展示在 timeline 里，不是弹框。用户期望默认用弹框模式。

**根因**: node-dev.md 第 89 行已经写了 `modal: true`：
```
defineWidgetStep({ modal: true, widget: SelectableTableWidget, ... })
```
但对话中 Agent 生成的代码没有加 `modal: true`。Agent 没有遵守指令。

**控制源**: node-dev.md 维度④的代码模板

**修复方案**: 
1. 在 node-dev.md 维度④中强调 `modal: true` 是默认必须项，不可省略
2. 提供完整的代码模板，减少 Agent 自由发挥空间：
```typescript
// 默认模板，modal: true 不可省略
defineWidgetStep({
  modal: true,
  widget: SelectableTableWidget,
  confirmedKey: '...',
  dialogTitle: '...',
  dialogSize: 'lg',
  ...
})
```

## 五、根本性设计原则问题

从 8 个 BUG 中提炼出 3 个根因：

### 原则 1: 能硬编码的别靠 Agent 自觉

node-dev.md 里**已经写了**"最后一步跳过维度④"、"modal: true"、"三行分别输出"，但 Agent 全部没遵守。自然语言描述规则，Agent 遵守率不可控。规则越多、越分散，遗漏率越高。

**应该怎么做**：能用代码约束的不要用文字约束。"最后一步跳过维度④"应该在 step-tracker.sh 硬编码；`modal: true` 应该是代码模板默认值。

**关联 BUG**: BUG-2, BUG-3, BUG-4, BUG-6

### 原则 2: 能一次性做的别分步做了再推翻

维度③先写自定义组件，维度④推翻重写。维度①写了 sort，维度②又问"怎么处理"。四维度的**执行时机**和**决策时机**错位。

**应该怎么做**：四个维度的问答只收集意图（不写代码），全部确认完毕后一次性生成代码。

**关联 BUG**: BUG-7, BUG-8

### 原则 3: 指令单源化——同一规则只定义一次

同一条指令在 node-dev.md、skill-guide.md、SKILL.md 里各写一遍，改一个漏一个，多个 agent 持有的"副本"逐渐分裂。

**应该怎么做**：规则只在一个地方定义（权威源），其他文件引用它。流程规则定义在 SKILL.md，agent 文件写"按 SKILL.md XX 章节执行"；代码模板放 lib/ 脚本，agent 调用脚本而非自己拼代码。

**关联 BUG**: BUG-2, BUG-3, BUG-6（指令写了但 agent 没遵守，部分原因是同一逻辑散落多处、表述不一致）

### 原则 4: 状态检测必须验证归属

检测到端口被占就认为 dev server 在跑，不验证是谁的进程。只看"是否存在"，不看"是否属于我"。

**应该怎么做**：端口检查验证 PID 的 cwd 是否是当前项目；数据恢复验证 userInput 是否已持久化。存在 ≠ 属于我。

**关联 BUG**: BUG-1, BUG-4
